# seaqt

`seaqt` is a set of `C` bindings for [Qt](https://www.qt.io/) generated by [`seaqt-gen`](https://github.com/seaqt/seaqt-gen), derived from the `C` part of [miqt](https://github.com/mappu/miqt). See `seaqt-gen` for more information.

The bindings are still going through significant changes, including in naming and structure - this repository is a preview intended for feedback.

The code may occasionally be **rebased** before the dust settles on a generally useful initial release.

## Using the bindings

`seaqt` can be used stand-alone or imported into other languages that support calling `C` functions via their [FFI](https://en.wikipedia.org/wiki/Foreign_function_interface).

### Qt versions

Bindings are generated for each Qt release into a separate branch named after the Qt `major.minor` version.

Bindings for a specific minor version can generally be used with all minor Qt versions following it - ie the `6.4` bindings are compatible with `6.5` etc but not with `6.2` and `7.0`. Use the lowest `minor` version that provides the functionality you need, for maximum compatibility!

| Qt Version | Branch |
| ---------- | ------ |
| 5.15+      | https://github.com/seaqt/seaqt/tree/qt-5.15 |
| 6.4+       | https://github.com/seaqt/seaqt/tree/qt-6.4  |
| 6.5+       | https://github.com/seaqt/seaqt/tree/qt-6.5  |
| 6.8+       | https://github.com/seaqt/seaqt/tree/qt-6.8  |
| 6.11+      | https://github.com/seaqt/seaqt/tree/qt-6.11 |

### Finding Qt

Qt gets located using `pkg-config` - use `PKG_CONFIG_PATH` to point to a specific Qt installation:

```sh
export PKG_CONFIG_PATH="/opt/Qt5.2.1/5.2.1/gcc_64/lib/pkgconfig/:$PKG_CONFIG_PATH"
```

### Include files

The generated C bindings are organized by Qt module, mirroring Qt's own module structure. Each module has its own directory under `seaqt/` (e.g. `seaqt/QtCore/`, `seaqt/QtWidgets/`). Within each module directory:

- One `gen_*.h` header per Qt C++ header file (e.g. `gen_qwidget.h`, `gen_qpushbutton.h`)
- One `gen_*.cpp` implementation file per header (compiled with a C++ compiler)
- An amalgamation file (e.g. `seaqt_QtWidgets.cpp`) that includes all module `.cpp` files for unity builds
- A `seaqt_QtWidgets.h` that includes all module `.h` headers

Every binding header includes `seaqt/libseaqt-runtime.h`, which provides shared types like `seaqt_string`, `seaqt_array`, and `seaqt_map` used across all modules.

To use a binding, include the corresponding header by module path:

```c
#include "seaqt/QtCore/gen_qobject.h"
#include "seaqt/QtWidgets/gen_qwidget.h"
#include "seaqt/QtWidgets/gen_qpushbutton.h"
#include "seaqt/libseaqt-runtime.h"
```

Each header uses an include guard of the form `SEAQT_<MODULE>_GEN_<HEADER>` to prevent multiple inclusion. Headers are `extern "C"` in C++ mode and expose opaque `typedef struct` declarations in plain C mode.

### Memory management

Objects are allocated and deallocated through explicitly named C functions. The generator tracks whether a C++ class has a public virtual destructor (the `CanDelete` flag); only those classes expose a `_delete` function.

**Creation:** Constructors become `_new` functions that return a pointer to the derived type. For classes with virtual methods, the constructor takes a vtable pointer and user data size:

```c
// Non-virtual class: simple constructor
QPushButton* QPushButton_new_text(const QPushButton_VTable* vtbl, size_t vdata, struct seaqt_string text);

// Base class constructor (e.g. QApplication)
QApplication* QApplication_new_int_char(const QApplication_VTable* vtbl, int vdata, int* argc, char** argv);
```

The vtable pointer enables polymorphism in C — callers define a static vtable instance for each subclass they create. The `vdata` argument reserves extra bytes after the C++ object for user-defined per-instance data (commonly used to store a pointer back to a host-language object).

**Destruction:** Classes with public destructors expose a `_delete` function:

```c
void QPushButton_delete(QPushButton* self);
```

Classes without a public destructor (e.g. utility classes with static factories) do not have a `_delete` function — they must be managed by their owning parent or through Qt's ownership model.

**Ownership model:** The bindings follow Qt's parent-child ownership model. When a widget is constructed with a parent, Qt owns the child and will delete it when the parent is destroyed. The `_delete` function is for explicit ownership transfer when no parent is set. In C, callers should manually call `_delete` for objects they own and that have no parent.

### Inheritance

The bindings simulate C++ inheritance using a vtable-based polymorphism scheme. Classes with virtual methods get a wrapper type named `Virtual<ClassName>` (e.g. `VirtualQPushButton`).

**Type layout:** A `VirtualXxx` instance contains the base object followed by a pointer to a vtable and user-defined data:

```
| -----------------------------------------------|
| sizeof(VirtualQObject) + alignment | vdata ... |
|------------------------------------|-----------|
| QObject | const QObject_VTable*    | ....      |
|------------------------------------|-----------|
```

**Vtable structure:** Each class with virtual methods gets a `ClassName_VTable` struct containing function pointers for each virtual method plus a destructor:

```c
typedef struct QObject_VTable {
    void (*destructor)(VirtualQObject* self);
    // ... one pointer per virtual method
} QObject_VTable;
```

**Usage pattern:** C callers define a static vtable instance and pass it to constructors:

```c
static QPushButton_VTable pb_vtbl = {
    .destructor = my_destructor,
    .mousePressEvent = my_mouse_press_event,
    // ... other virtual methods as NULL
};

struct MyWidget {
    VirtualQPushButton* button;
    int custom_data;
};

struct MyWidget env = {
    .button = QPushButton_new_text(&pb_vtbl, sizeof(struct MyWidget),
                                   seaqt_literal("Hello"))
};
```

**Type casting:** Base pointers are obtained via `_virtbase` functions that handle virtual inheritance pointer adjustment:

```c
void QPushButton_virtbase(QPushButton* src, QWidget** outptr_QWidget, QObject** outptr_QObject);
```

**Vdata access:** Helper functions convert between the raw `void*` user data area and the derived type:

```c
void* VirtualQPushButton_vdata(VirtualQPushButton* self);
VirtualQPushButton* vdata_VirtualQPushButton(void* vdata);
```

### Signals and slots

Signal connections use a callback-based mechanism that bridges C++ `QObject::connect` with C function pointers. Each signal gets a dedicated `connect` function:

```c
void QPushButton_connect_pressed(
    QPushButton* self,
    intptr_t slot,                     // User data passed to callback
    void (*callback)(intptr_t),        // The C callback
    void (*release)(intptr_t)          // Cleanup callback (can be NULL)
);
```

**Parameters:**
- `slot`: An `intptr_t` value passed as the first argument to the callback. This is typically a pointer to user-defined data cast to `intptr_t`.
- `callback`: A C function that receives the signal parameters. Its signature includes `intptr_t` as the first parameter followed by the signal's typed parameters.
- `release`: An optional cleanup function called when the connection is destroyed. Pass `NULL` if no cleanup is needed.

**Example:**

```c
struct HelloWorld {
    VirtualQPushButton* button;
    int counter;
};

void onPressed(intptr_t data) {
    struct HelloWorld* env = (struct HelloWorld*)data;
    env->counter++;
    QAbstractButton_setText((QAbstractButton*)env->button,
                           seaqt_literal("Clicked!"));
}

// Connect:
QAbstractButton_connect_pressed(
    (QAbstractButton*)env.button,
    (intptr_t)&env,
    onPressed,
    NULL  // no cleanup needed
);
```

For Qt 6.9+, private signals (declared in the `private slots:` section of headers) are also exposed through the same `connect` mechanism.

## wip

The bindings are a preview and still undergo significant changes. Current areas of active development:

* **Naming stability:** Overload disambiguation uses a multi-strategy scheme (parameter names → type names → cv-qualification). New overloads can cause existing symbols to be renamed.
* **Qt version coverage:** Bindings target Qt 5.15, 6.4, 6.5, 6.8, and 6.11. Older Qt 6.x versions (e.g. 6.2, 6.3) are not directly supported; use the lowest minor version that provides needed functionality for maximum compatibility.
* **Platform restrictions:** Some classes and methods have platform guards (e.g. Android exclusions) embedded in the generated code.
* **Missing bindings:** Not every Qt API is exposed — some are blocked by complex types, unsupported patterns, or ABI concerns. The allowlist in `seaqt-gen/cmd/genbindings/config-allowlist.go` controls what gets generated.
* **Build integration:** No official build system integration (CMake, Meson, etc.) is provided yet. Projects must manually compile the generated `.cpp` files and link against Qt.
